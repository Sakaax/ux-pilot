# Churn Prevention Rules

## Exit intent
- Detecter l'intention de depart (curseur vers la fermeture, scroll rapide)
- Proposer une offre de retention (discount, feature unlock, aide)
- Ne pas bloquer — proposer, pas forcer

## Downgrade avant cancel
- Proposer un downgrade avant l'annulation complete
- "Passez au plan gratuit plutot que d'annuler"
- Montrer ce que l'utilisateur perd vs ce qu'il garde

## Feedback obligatoire
- Avant le depart : demander la raison (obligatoire mais court)
- Choix multiples + champ libre
- Utiliser le feedback pour ameliorer le produit

## Win-back flow
- Email sequence post-churn (J+1, J+7, J+30)
- Offre speciale de retour si pertinent
- Montrer les nouvelles features depuis le depart

## Pause option
- Permettre de mettre le compte en pause plutot qu'annuler
- "Prenez une pause de 1-3 mois, vos donnees sont preservees"
- Moins definitif que cancel, plus facile de revenir

## Cancel flow
- Le bouton cancel ne doit pas etre cache (dark pattern)
- Process clair : raison → offre retention → confirmation
- Confirmation explicite ("Oui, annuler mon abonnement")
- Email de confirmation d'annulation

## Data export
- Permettre l'export des donnees avant le depart
- L'utilisateur doit pouvoir partir avec ses donnees
- Ca construit la confiance meme si certains restent grace a ca
